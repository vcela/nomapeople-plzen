This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A marketing/registration site (Czech language) for a summer 2026 contact improvisation dance series in Plzeň, run under the "NŌMA people" brand. The public page (`index.html`) is a static, no-build-step single-page document; it POSTs to a small classic PHP + MariaDB backend (`api/`, `admin/`, `inc/`, `lib/`) for registration storage, email, and an admin panel. The whole thing is meant to run unmodified on classic shared PHP hosting (Webglobe) — no build step, no Composer, no framework anywhere in the repo.

- **Live URL**: `https://nomapeople.com/plzen/`
- **Content**: hero, "what is contact improvisation", practical info (dates/location/price), teacher bio, FAQ, map, a registration form that now actually registers people, and a GDPR/privacy page (`zasady-ochrany-osobnich-udaju.html`, plain static HTML, controller identity hardcoded there as Martina Matoušková / no IČO / `maya@nomapeople.com` — update that file directly if these details change).

## Architecture: this is a "Design Component" (DC) document, not plain HTML

`index.html` is not a normal static page — it's authored in a custom template DSL rendered client-side into React by `support.js`. Understanding this is essential before editing:

- Everything the page renders lives inside a single `<x-dc>...</x-dc>` block. Regular tags (`div`, `section`, etc.) are used, but a few custom tags drive logic:
  - `<sc-for list="{{ expr }}" as="item">` — loop
  - `<sc-if value="{{ expr }}" hint-placeholder-val="{{ ... }}">` — conditional render
  - `{{ expr }}` — interpolation inside text or attributes, resolved against the values returned by the component's `renderVals()`
  - `<helmet>...</helmet>` at the top of `<x-dc>` — becomes real `<head>` tags (title, meta, styles) via `support.js`'s helmet manager
- The trailing `<script type="text/x-dc" data-dc-script>` block defines `class Component extends DCLogic { ... }`. This is the page's logic: React-style `state`, lifecycle methods (`componentDidMount`, `componentWillUnmount`), event handlers, and a `renderVals()` method that returns the flat object all `{{ }}` expressions in the template resolve against. This is effectively a class-component React file inlined as a string and `eval`'d at runtime — treat it with the same care as real React source (correct `this` binding via class fields/arrow functions, clean up listeners/timers in `componentWillUnmount`, etc.).
- `data-props` JSON on the `<script data-dc-script>` tag declares the component's external props (here: `formMode`, an enum of `"inline" | "google"`) — but its `"default"` field is **editor-only metadata**, read by `runtime.adoptParsed()` into `propsMeta` for a design-tool UI. It is *not* applied to `this.props` when the page boots standalone (no CMS/preview iframe pushing `propOverrides` via `__dcSetProps`) — in that case `getDC()`'s `Dispatcher` renders with `entry.propOverrides || null`, i.e. `this.props` ends up `{}`. **The real, live-site default is whatever the logic class itself falls back to** — here `const mode = this.props.formMode ?? 'inline';` in `renderVals()`. If you change which mode/variant should show by default on the actual deployed page, change *that* fallback, not (or not only) the `data-props` JSON — editing only the JSON default is a no-op for anyone opening the page directly.
- **`support.js` is generated output** — the file header says `// GENERATED from dc-runtime/src/*.ts — do not edit. Rebuild with 'cd dc-runtime && bun run build'.` The `dc-runtime` source is not part of this repo checkout, so `support.js` should be treated as a vendored, read-only runtime. Never hand-edit it; all real content/behavior work happens inside `index.html`'s `<x-dc>` template and `<script data-dc-script>` logic block.
- **`{{ expr }}` expressions do not support ternaries or arbitrary JS** — `resolve()` in `support.js` only understands identifier/property paths, `===`/`!==`/`==`/`!=`, `!expr`, literals, and parens. Something like `{{ submitting ? 'a' : 'b' }}` silently fails to parse as intended (it's not a syntax error — it just doesn't resolve the way you'd expect). Compute any conditional/derived string in the logic class's `renderVals()` instead (see `submitLabel`/`submitOpacity` next to the FAQ `icon`/`mh` fields for the existing pattern) and interpolate the plain resulting value with `{{ someVar }}`.

## Editing the page

- All styling is inline `style="..."` attributes (no external CSS file, no CSS classes for one-off styling) plus a small `<style>` block in `<helmet>` for global rules, `@keyframes`, and responsive breakpoints (`@media` at 960px/767px). Follow this pattern — don't introduce a separate stylesheet or CSS framework.
- Design tokens (colors, layout widths) are set as CSS custom properties in `:root` inside the `<style>` block (`--shell-max-width`, `--header-h-desktop`, etc.) and as literal hex values reused throughout inline styles (`#884858` wine/primary, `#8fb6c7` blue accent, `#f4ecc6` cream, `#ece5d8` background, `#2a2320` dark text/footer). Reuse these exact values rather than introducing new colors.
- `style-hover="..."` and `style-focus="..."` are custom pseudo-class attributes handled by the DC runtime's `pseudoClass` mechanism (compiled into real `:hover`/`:focus` stylesheet rules) — this is how hover/focus states are declared since real inline styles can't express pseudo-classes.
- Section anchors (`#o-ci`, `#info`, `#lektorka`, `#faq`, `#prihlasit`) are targeted both by the header/footer nav links and by the scrollspy logic in `componentDidMount` (`data-spy` attributes) — if you rename or add sections, keep the `id`, the nav `data-spy` value, and any anchor links in sync.
- `data-reveal` marks elements for the scroll-triggered fade/slide-in animation set up in `componentDidMount`; add it to new top-level content blocks to keep the reveal effect consistent, or omit it for elements that should render immediately.
- The registration form (`#prihlasit`) has two mutually exclusive modes controlled by the `formMode` prop: an inline form (`onSubmit` in `index.html`, fast client-side checks then `fetch('api/register.php', ...)`) or a link out to an external Google Form (`googleUrl`). Only one is shown at a time via `showInline`/`showGoogle`. The inline path is the live one — client-side validation is a fast-fail UX layer only; `api/register.php` re-validates everything server-side and is the actual source of truth.
- The nav's "Administrace" link (`<sc-if value="{{ isAdmin }}">`) is driven by a `fetch('api/session.php')` call in `componentDidMount` — it's the only way the static page learns whether a browser has an authenticated admin session.
- `assets/` holds images referenced by relative path (`assets/*.png`, `assets/*.webp`, `assets/favicon.svg`). `googlee1c50f212493e90d.html` is a Google Search Console site-verification file — leave it untouched.

## Cookie consent (assets/consent.js)

- **Never add an unconditional analytics/marketing `<script>` tag anywhere in this repo.** All tracking is gated behind explicit visitor consent, loaded by `assets/consent.js` — a small dependency-free vanilla-JS file, deliberately *not* part of the DC/React component tree, shared between `index.html` and `zasady-ochrany-osobnich-udaju.html` (the GDPR page, plain static HTML).
- It's included directly in `index.html`'s real `<head>` (not inside `<helmet>`) so the banner appears — and analytics stays correctly gated — even before/if the React CDN load the rest of the page depends on finishes or fails. Don't move it into `<helmet>`; content there only reaches the real `<head>` after the DC runtime's first React render.
- Consent decision lives in `localStorage` (`nm_cookie_consent`: `'accepted' | 'rejected'`, unset = not yet decided → banner shows). Accepting calls `loadGoogleAnalytics()` (GA4, measurement ID `G-XZS41C1ELT`) and `loadMetaPixel()`. `META_PIXEL_ID` is currently an empty-string placeholder in `consent.js` — Meta Pixel is a deliberate no-op until a real Pixel ID is filled in there.
- `window.nmOpenCookieSettings()` clears the stored decision and re-renders the banner — this is what the footer's "Nastavení cookies" button (`index.html`, wired through `openCookieSettings` in the logic class since footer markup goes through the DC template compiler and needs a real function reference, not an inline `onclick="..."` string) and the GDPR page's "Změnit nastavení cookies" button both call.
- No jsdom/browser automation is available in this environment; `consent.js` was verified with a small hand-rolled DOM/localStorage mock run through Node's `vm` module (see chat history) rather than skipped — extend that approach rather than assuming behavior when touching this file.

## Backend: PHP + MariaDB (api/, admin/, inc/, lib/)

Classic shared-hosting PHP — no Composer, no framework, no autoloader. Structure:

- `config.sample.php` (committed, placeholders) → copy to `config.php` (gitignored) with real DB/SMTP credentials before deploying. Every entry point (`api/*.php`, `admin/*.php`) requires it directly via `__DIR__ . '/../config.php'`.
- `inc/db.php` — single `db(): PDO` factory (utf8mb4, exceptions, real prepares — no `PDO::ATTR_EMULATE_PREPARES`).
- `inc/auth.php` — session bootstrap (`auth_bootstrap()`) plus the "remember me" system: every successful admin login issues a persistent cookie automatically (no opt-in checkbox), using a selector/validator split-token pattern stored in `admin_remember_tokens` (only a sha256 hash of the validator is ever stored). Tokens rotate on every use; a validator mismatch on a known selector is treated as a stolen cookie and wipes every token for that admin (fail closed). `current_admin()` / `require_admin()` are what every protected page calls.
- `inc/csrf.php` — session-bound CSRF tokens, used on every authenticated `admin/*.php` POST form (including the logout button in `admin/inc/header.php`). `api/register.php` deliberately has no CSRF token — it has no session/ambient authority to protect, so a token there would be theater; it instead does a cheap same-`Origin`-host check as defense-in-depth.
- `inc/mailer.php` — wraps vendored PHPMailer (`lib/PHPMailer/src/`, pinned release, required via plain sequential `require_once`, no autoloader) for the admin notification email (recipient address is the `notification_email` row in the `settings` table, editable at `admin/settings.php`) and the registrant confirmation email. Both send functions catch everything and `error_log()` on failure — a bounced/misconfigured mailbox must never undo an already-stored registration or break the JSON contract of `api/register.php`.
- `api/register.php` (public, POST) and `api/session.php` (public, GET, `Cache-Control: no-store`, returns `{isAdmin: bool}` only — never who) are the only endpoints the public page talks to.
- `admin/` is plain server-rendered PHP+HTML (not the DC/React template system) styled with the same color tokens as the public page. `admin/inc/header.php`/`footer.php` are shared chrome; each protected page calls `require_admin()` itself *before* including the header, so an unauthenticated hit never renders the shell. `admin/users.php` is add+list only (no delete) and `admin/account.php` (change own password) revokes every other session's remember token on a successful change — by design, not a bug, if you're chasing a "why did my other tab log out" report.
- **Always redirect within `admin/` via `admin_redirect($file)` (`inc/helpers.php`), never a bare `header('Location: some.php')`.** A relative Location header resolves against the request URL's directory, and for a bare directory hit with no trailing slash (`/admin`, not `/admin/` — which is exactly what `require_admin()` sees on first visit) that directory is `/`, not `/admin/`, so a relative redirect silently 404s at the domain root instead of reaching `/admin/login.php`. `admin_redirect()` builds an absolute path from `$_SERVER['SCRIPT_NAME']` instead, which is correct regardless of trailing slash or deployment subpath (`/admin/...` locally vs `/plzen/admin/...` live).
- `admin/index.php` groups registrations by *lesson week*, not by calendar week: `lesson_bucket_date()` maps a row's `created_at` to the next occurrence (on-or-after, so a same-day signup belongs to today's lesson) of the `lesson_weekday` setting (ISO 1–7, editable at `admin/settings.php`, `czech_weekday_name()` in `inc/helpers.php` renders it). Groups are sorted soonest-lesson-first (`krsort` on the `'Y-m-d'` bucket key — string sort works because that format is lexically ordered). "Delete" on a row is a **soft delete only** — it toggles `registrations.deleted_at` (POST `action=toggle_delete` handled at the top of `admin/index.php`, before any output, then redirects) and is fully reversible from the same button; deleted rows stay visible, struck through (`.row-deleted`), sorted to the end of their group, and excluded from that group's headline count.
- `schema.sql` (repo root) is the only source of truth for the DB shape — there is no migration tooling. Run it by hand via phpMyAdmin against a fresh database. It's idempotent (`IF NOT EXISTS` / `ON DUPLICATE KEY UPDATE`, plus MariaDB's `ADD COLUMN IF NOT EXISTS` for columns added after a database already exists) so re-running it against an already-provisioned database is safe and won't clobber values an admin has since changed (e.g. `notification_email`).

## Testing changes

No build/lint/test tooling (no `package.json`, no Composer). For `index.html`: open it in a browser or serve the directory statically — `support.js` bootstraps React from CDN (`unpkg.com`), so an internet connection is required and changes are visible on a simple reload. For the PHP backend: `php -l` catches syntax errors; a real end-to-end check needs a MariaDB instance — locally this machine has WampServer installed (`c:\wamp64\bin\mariadb\...\bin\mariadbd.exe`, data dir configured in the adjacent `my.ini`; the `wampmariadb64` Windows service exists but needs an elevated shell to start/stop). Point a throwaway `config.php` at a scratch database, load `schema.sql` into it, run `php -S 127.0.0.1:8000`, and exercise the endpoints with `curl` (use `--data-binary @file` for any payload containing non-ASCII/Czech text — passing UTF-8 as a raw `-d`/`--data-urlencode` shell argument through Git Bash on Windows can mangle the bytes before curl.exe ever sees them). Delete the throwaway `config.php` afterward — never leave real or scratch DB credentials sitting in the repo root.
